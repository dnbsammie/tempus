<h1 id="contributing">🤝 Contributing</h1>

<p>
I greatly appreciate your contributions to this project. To maintain order and clarity, always use the <strong>predefined templates</strong> both Pull Requests (PR) and Issues.
</p>

<h2>✅ Good practices</h2>

<h3>⌨ Commit convention</h3>
<p>This project follows the convention of <a href="https://www.conventionalcommits.org/">Conventional Commits</a>:</p>

<ul>
  <li><strong>feat:</strong> New functionality.</li>
  <li><strong>fix:</strong> Bug fixes.</li>
  <li><strong>docs:</strong> Changes to documentation.</li>
  <li><strong>style:</strong> Formatting or design (no changes to logic).</li>
  <li><strong>refactor:</strong> Refactoring without functional changes.</li>
  <li><strong>test:</strong> Added or modified tests.</li>
  <li><strong>chore:</strong> Minor tasks (build, dependencias, etc.).</li>
  <li><strong>release:</strong> release of a new version.</li>
  <li><strong>BREAKING CHANGE:</strong> New functionality</li>
</ul>

<h4>Example:</h4>

<p><strong>Title:</strong> <code>feat(api):</code> add endpoint to create users</p>
<p><strong>Description:</strong> Explain what your commit add, do and solve.</p>

<h3>📌 How to send a Pull Request?</h3>
<p>If you want to propose a change, follow these steps. For new features, always use a branch called <code>features/feature-name</code>.</p>

<ol>
  <li><em>Fork</em> the repository.</li>
  <li>Create a new branch following the convention: <code>features/feature-name</code>.</li>
  <li>Before sending, run <code>git pull origin main</code> to synchronize the latest changes.</li>
  <li>Open a Pull Request using the template <code>pull_request_template.md</code>. Don't omit the template.</li>
  <li>Complete all required fields: description/context, exchange rate, and checklist</li>
</ol>

<h3>🐞 How to report an issue?</h3>
<p>To maintain organization, use the templates available in the <code>issues</code> branch. When creating a new issue on GitHub, select the appropriate template:</p>

<ul>
  <li><strong>Bug Report:</strong> To report errors or bugs in the code.</li>
  <li><strong>Feature Request:</strong> To propose new features or improvements.</li>
  <li><strong>Question:</strong> To ask questions or make inquiries related to the project.</li>
</ul>
<p>Each template includes specific sections that you must complete (description, steps to reproduce, environment, etc.). This helps maintain clarity and facilitates review.</p>

<h2>📜 Rules</h2>

<ol>
  <li>Write clear and concise descriptions in your PRs and Issues.</li>
  <li>Double-check that your code compiles and passes the tests before submitting.</li>
  <li>Maintain a consistent style with the rest of the project.</li>
  <li>Maintain a consistent style with the rest of the project.</li>
</ol>

<p>This project values ​​clarity, simplicity, and long-term maintainability. Please be kind and professional in all your interactions.</p>
